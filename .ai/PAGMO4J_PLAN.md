# pagmo4j — Java/Kotlin Port Plan

## Naming

**`pagmo4j`** — follows JVM ecosystem convention (log4j, gson, etc.), clearly means "pagmo for Java", works equally well as a Kotlin library. Maven coordinates: `io.github.samthegliderpilot:pagmo4j`. Published to Maven Central.

## Layer 1 — SWIG interface files (~70% reuse)

The language-neutral skeleton is unchanged: `%module`, `%include`, `%feature("director")`, `%extend`, `%ignore`, `%rename` all work identically for `-java`.

**What must be replaced** is the 91 `%typemap(cs*)` directives:

| C# directive | Java equivalent |
|---|---|
| `%typemap(csclassmodifiers)` | `%typemap(javaclassmodifiers)` |
| `%typemap(cstype)` | `%typemap(jstype)` / `%typemap(jtype)` |
| `%typemap(csdirectorout)` | `%typemap(javadirectorout)` |
| `%typemap(csin)` | `%typemap(javain)` |
| `%typemap(csout)` | `%typemap(javaout)` |

This is a mechanical find-replace pass over the `.i` files, with some per-typemap logic adjustments (e.g., `ulong` → `long`, `IntPtr` → `long` for pointer handles).

**Key difference:** C# uses `partial class` to split generated and handwritten code. Java has no partial classes. The solution is to use SWIG's `%extend` directive to inject helper methods directly into the generated class — this is actually cleaner than partial classes and keeps the additions in the `.i` file where they belong.

## Layer 2 — Handwritten extension files (rewrite, mostly mechanical)

The 79 `.cs` files fall into clear categories:

**Trivial mechanical translation (~40 files):**
Algorithm/problem extension files (`de.cs`, `rosenbrock.cs`, etc.) add typed log projections and helper methods. These translate almost 1:1: C# properties → Java getters, LINQ → Java streams, `params double[]` → Java varargs.

**Pattern substitutions (~25 files):**

| C# pattern | Java equivalent |
|---|---|
| `IDisposable` + `using` | `AutoCloseable` + try-with-resources |
| `partial class` additions | `%extend` in `.i` files OR Kotlin extension functions |
| `ConditionalWeakTable<K,V>` (GC rooting) | `java.util.WeakHashMap` or `java.lang.ref.Cleaner` |
| `GCHandle` (pinning callbacks) | JNI global refs (SWIG handles this automatically) |
| `NativeLibrary.TryGetExport` | `System.loadLibrary` + JNI symbol lookup |
| `[DllImport]` P/Invoke | JNI (generated by SWIG automatically) |

**Genuinely new work (~14 files):**
- `NativeInterop.cs` — the GC-rooting and callback ownership machinery. The JNI model handles most of this automatically (SWIG directors manage JNI global refs), but the `ConditionalWeakTable` rooting strategy needs rethinking using `java.lang.ref.WeakHashMap` or Java 9+ `Cleaner`.
- `ProblemHandle.cs` (SafeHandle) → Java doesn't have `SafeHandle`; use a simple wrapper with `Cleaner` or `PhantomReference`.
- `managed_thread_bfe.cs` → replace `Parallel.For` + `ThreadLocal<T>` with `ExecutorService` + `ThreadLocal<T>`. Pattern is identical conceptually.
- `IThreadCloneableProblem.cs` + `ExclusiveCloneAdapter` → Java interface + inner class. Trivial.

## Layer 3 — Build and packaging

**Gradle** (not Maven) for modern Java/Kotlin projects. Kotlin support is first-class in Gradle.

Native library bundling:
- SWIG generates a JNI `.so`/`.dll`/`.dylib` (same `PagmoWrapper` native library, different JNI entry points)
- Bundle in JAR under `natives/` with a small runtime loader that extracts and loads the right binary for the OS/arch
- Publish to Maven Central as a fat JAR (natives bundled) — same model as lwjgl, sqlite-jdbc, etc.

**Module name:** `io.github.samthegliderpilot.pagmo4j` (Java 9+ module system)

---

## Kotlin angle

Kotlin is a first-class citizen at zero extra cost:
- SWIG generates Java; Kotlin consumes Java directly
- The `partial class` workaround (using `%extend` in `.i` files) means all helpers are on the Java types, which Kotlin calls naturally
- `IThreadCloneableProblem` becomes a Kotlin `interface` with a default `clone()` returning `null`
- `AutoCloseable` integrates with Kotlin's `use { }` block (exact equivalent of C# `using`)
- Kotlin extension functions can add any C#-style PascalCase helpers on top of the Java types without touching the generated code

The Kotlin experience would be essentially identical to the C# one.

---

## Recommended structure

Separate repository (`pagmo4j`) rather than monorepo — the native binary is shared (same `PagmoWrapper` build), but the JVM binding is an independent project with its own build system, test suite, and Maven Central publication.

The shared native layer (`pagmoWrapper/`) lives in the Pagmo.NET repo and is consumed by `pagmo4j` as a pre-built binary downloaded from GitHub Releases, avoiding the need to build C++ in both repos.

## What's genuinely harder in Java vs C#

1. **No partial classes** — `%extend` in `.i` files is the cleanest workaround but means the `.i` files grow. Alternatively, Kotlin extension functions cover the gap elegantly.
2. **GC non-determinism** — Java's GC is even less predictable than .NET's (no `GC.AddMemoryPressure` equivalent until Java 21's Foreign Function & Memory API). The `try-with-resources` discipline is more important in Java.
3. **`size_t` mapping** — Java has no unsigned types. `size_t` maps to `long` (same as C# `ulong`), but the `SizeTInterop` conversion layer needs equivalent overflow guards.
4. **Android** — if Android support is desired, the native build needs ABI splits (arm64-v8a, x86_64). This is a separate concern but the JNI model supports it naturally.
